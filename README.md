# Коррекция дрейфа VIO по опорной точке (ArduCopter + SITL, без GPS)

Решение тестового задания: квадрокоптер летит без GPS по смоделированному VIO,
оценка позиции накапливает дрейф, оператор «отбивает» опорную точку с известными
координатами — накопленный дрейф убирается без опасного скачка в полёте.

## Запуск с нуля
Стенд работает в WSL (Ubuntu 24.04). Нужны две консоли WSL и Mission Planner
на Windows: готовая кастомная сборка — папка `MissionPlanner/MissionPlanner_net461`
этого репозитория, запускать `MissionPlanner.exe` из неё.

**Окно 1 — SITL** (при первом запуске скрипт соберёт его сам, ~несколько минут):

```bash
cd ~
git clone --recurse-submodules https://github.com/ArduPilot/ardupilot.git
./ardupilot/Tools/environment_install/install-prereqs-ubuntu.sh -y
git clone https://github.com/VermennikovMV/Test_Task_04.07.git
cd Test_Task_04.07/SITL && bash sitl_launch.sh
```

**Окно 2 — источник VIO:**

```bash
source ~/venv-ardupilot/bin/activate
cd ~/Test_Task_04.07/vio-sim && python3 vio_launch.py
```

**Mission Planner:** Connect → TCP → `127.0.0.1` → порт `5762`.

**Маркер A** (опорная точка) зашит в конфиге: 150 м севернее и 200 м восточнее
точки старта. Чтобы видеть его на карте, поставить POI: правый клик →
Point of Interest → Add POI → координаты `-35.3619146;149.1674405`.

**Полёт:** режим Guided → Arm → правый клик по карте → Takeoff (10 м) →
Fly to here, пару минут копим дрейф → подводим дрон так, чтобы красный
призрак SIM (истина) встал на POI → в окне 2 команда `snap A`
(всё как в видео-демонстрации).

## Отчёт (п.5 ТЗ)

### 1. Как VIO подаётся в EKF3 и как задаётся origin без GPS

Позиция и курс отправляются сообщением `VISION_POSITION_ESTIMATE` (30 Гц,
MAVLink2 — поля `covariance` и `reset_counter` являются расширенными и в первой
версии протокола молча отрезаются). Приёмный тракт ArduPilot:
`GCS_MAVLink -> AP_VisualOdom -> EKF3`.

<details>
<summary><b>Подробно</b> — путь сообщения внутри автопилота и математика коррекции</summary>

Путь сообщения: VISION_POSITION_ESTIMATE → AP_VisualOdom (VISO_TYPE=1) — извлекаются позиция, углы, ковариация и reset_counter, применяются поправки монтажа (VISO_ORIENT/POS), учитывается задержка (VISO_DELAY_MS) → измерение кладётся в буфер EKF3 со своей меткой времени.

Математика коррекции — на трёх величинах: P — неуверенность фильтра в собственном состоянии (растёт на каждом такте предсказания по IMU, падает при принятии измерения от VIO), R — заявленная шумность измерения (у нас — ковариация из сообщения: σ по позиции, крену/тангажу и курсу), K = P/(P+R) — коэффициент доверия.

На такте слияния EKF3 считает инновацию (измерение − предсказание) и проверяет её гейтом относительно суммарной неопределённости S = P+R. Если прошло — состояние поправляется на K·инновация (через ковариационные связи поправка позиции распределяется также на скорость, углы и смещения датчиков), после чего P уменьшается. В штатном полёте P ходит туда-сюда: прогноз — вверх, коррекция — вниз, около равновесия. При потере VIO остаются одни прогнозы — P растёт неограниченно, что и приводит к EKF failsafe.

Особый случай — скачок потока при отбивке: изменившийся reset_counter сообщает EKF, что источник перескочил штатно (дрон физически не телепортировался). Вместо боя с гейтом выполняется ResetPositionNE: оценка мгновенно перескакивает, P по позиции выставляется под неопределённость источника, а контроллеру позиции передаётся вектор скачка — он сдвигает цель на ту же величину, ошибка регулятора не меняется, и физического рывка не происходит.

</details>

Источники EKF3 (файл `vio_params.parm`, грузится при старте SITL — reboot не нужен):

```
EK3_SRC1_POSXY=6 (ExternalNav)   позиция — наш VIO
EK3_SRC1_VELXY=0                 скорость не подаём (см. «ограничения»)
EK3_SRC1_POSZ=1  (Baro)          высота — барометр
EK3_SRC1_VELZ=0                  вертикальная скорость: прямого источника нет,
                                 EKF выводит её сам (аксель + баро через ковариации);
                                 дефолт GPS при GPS_TYPE=0 валит арминг-проверку
                                 «EK3 sources require GPS»
EK3_SRC1_YAW=6                   курс — наш VIO; компасы выключены: курс и позиция
                                 из ОДНОГО мира, «двум северам» неоткуда разойтись
VISO_TYPE=1 (MAVLink)            «доверчивый» бекенд: источник сам отдаёт данные
                                 в мире EKF. Бекенды 2/3 (T265/VOXL) — драйверы под
                                 капризы конкретных камер.
```

**Origin.** Без GPS EKF не знает точку отсчёта — мост отправляет
`SET_GPS_GLOBAL_ORIGIN` (точка старта SITL). Нюанс: EKF принимает origin 
только после инициализации фильтра (первые секунды после
старта — отказ), поэтому мост шлёт с повторами и ждёт подтверждения-эха
`GPS_GLOBAL_ORIGIN`. **Home** без GPS сам не выставляется (автоустановка привязана
к GPS-фиксу), а без home коптер не армится («AHRS: waiting for home») — мост
ставит его следом через `COMMAND_INT / MAV_CMD_DO_SET_HOME` с проверкой ACK.

### 2. Как смоделирован VIO, включая модель дрейфа

Моделируются не сырые датчики, а их итоговый эффект на выходе одометрии. Теоретическая
опора: у VIO ровно четыре ненаблюдаемых степени свободы — глобальные X, Y, Z
и курс; крен и тангаж прибиты гравитацией и не дрейфуют — они не моделируются
и отправляются истинными.

Модель построена по этажам датчиков (каждый член — след своего физического
источника), тик 30 Гц, `dt = 1/30`:

```
a_ost  = N(0, σ_a)          остаток ошибки акселерометра в этом тике: то, что
                            «камера» внутри VIO не доела из биаса; в среднем ноль
v_err += a_ost * dt         интеграл №1 (ускорение -> скорость): «ложная скорость»
p_err += v_err * dt         интеграл №2 (скорость -> позиция): гладкий снос,
                            растёт быстрее линейного — как настоящая инерциалка

w_ost  = N(0, σ_w)          остаток ошибки гироскопа по курсу (один интеграл)
psi   += w_ost * dt         накопленная ошибка курса

p_vio  = Rz(psi) @ p_true + p_err + N(0, σ_n)
yaw_vio = yaw_true + psi
```

Подробнее в модуле vio_data.py

### 3. Как устроена «отбивка» и почему коррекция безопасна

Ключевое проектное решение: **дрейф — свойство источника, поэтому лечится
в потоке, а не в EKF**. Разовая подача «истинного» измерения в EKF проигрывает
всегда: следующие 30 дрейфующих сообщений в секунду либо утащат оценку обратно
(малый дрейф), либо не пройдут гейт инноваций и вызовут остановку принятия VIO
вплоть до EKF failsafe (большой дрейф). Поток побеждает разовое
сообщение — чинить надо поток.

Отбивка (оператор навёл дрон над маркером и скомандовал `snap A`):

```
Δ = последняя_отправленная_позиция − координаты_маркера    (накопленный дрейф)
offset += Δ            offset вычитается из ВСЕХ последующих сообщений
reset_counter += 1     «скачок потока легален»
```

Безопасность обеспечивает **штатный** механизм ArduPilot для скачков источника:
изменение `reset_counter` → EKF выполняет мгновенный `ResetPositionNE`
и оставляет контроллеру позиции «записку» о векторе скачка →
контроллер сдвигает свою цель на тот же вектор → ошибка PID не меняется →
оценка прыгает, дрон физически не уехал. Дальше по режимам: Loiter —
цель остаётся переименованной (держим, где висим); Auto/Guided — сдвиг уходит
во временный буфер и плавно списывается: дрон плавно доезжает до настоящей
точки маршрута. Записка предотвращает рывок и движение по ложной причине.

### 4. Поведение при неточной коррекции, повторах и потере VIO

- **Неточное наведение** — обрабатывается конструкцией, а не кодом: промах
  оператора записывается в `offset` и становится остаточной ошибкой, равной ровно
  промаху, следующая отбивка добирает остаток. Порогов на величину поправки сознательно нет:
  скачок любого размера безопасен (reset_counter легализует его без рывка),
  а грубый промах не копится и снимается следующей отбивкой.
- **Повторные коррекции** — Δ каждой новой отбивки считается
  от уже исправленного выхода, т.е. равна только свежему дрейфу. При частых коррекциях
  дельта будет просто примерно равной 0.
- **Потеря VIO** (`loss N`) — EKF летит на инерциалке, в P-матрице растут дисперсии. При
  возобновлении `reset_counter` легализует накопившийся скачок.

### 5. Известные ограничения и что бы я улучшил
  В будущем можно также оттестировать всё уже на настоящем OpenSource VIO в Gazebo (то есть моделировать визуальную навигацию через симулятор). По граничениям: сейчас ковариации от моей модели VIO зашитые и не меняются с накопленным дрейфом, данный вопрос можно изучить и попробовать менять неуверенность с дрейфом, чтобы модель стала более точной. Кроме того, коррекции по курсу (можно маркеру задать также курс, тогда отбивка курса симметрична позиционной:  `offset_yaw += последний_отправленный_yaw − yaw_маркера`, далее каждый тик `yaw_выход = yaw_VIO − offset_yaw`) и высоте не применяются. Также можно улучшить модель VIO, чтобы она посылала в том числе скорость (продифференцировав позицию, например — скорость обязана быть согласованной с врущей позицией, а не честной), соответственно посылать скорость в EKF, тем самым улучшив немного полёт. Можно ещё придумать то, какой Failsafe action совершать при долговременной потери связи с VIO.
